# AI 应用开发 Skill

## 目标

帮助 AI 编程助手完成 AI 应用从需求理解、方案设计、模型与架构选择、代码实现、评测验证到交付说明的完整闭环。重点不是“能调通模型”，而是交付一个可维护、可评估、可降级、成本和安全可控的 AI 能力。

## 核心原则

1. 先明确用户价值和失败边界，再选择模型、Prompt、RAG、Agent 或工具调用方案。
2. 先读取现有代码、依赖版本、配置和已有 AI 封装，再新增 SDK、模型或抽象。
3. 第三方 AI 服务必须按官方文档和项目实际版本实现；不要凭记忆调用 API。
4. Prompt、检索、工具调用、后处理和业务规则要分层清晰，避免把所有逻辑塞进一段 Prompt。
5. AI 输出必须有边界：校验结构、处理空结果、拒答、超时、限流、部分失败和不可置信结果。
6. 对用户可见的 AI 能力必须设计评测方式；不能只用一次手工样例判断效果。
7. 评估成本、延迟、并发、上下文长度、Token 预算、日志脱敏和隐私数据边界。
8. 生产链路要支持观测、回放、灰度、回滚和降级，不把模型结果当作绝对事实。

## 与其他 Skill 的关系

- 本 skill 负责 AI 链路架构、效果边界、评测、Prompt/RAG/Agent 质量和 AI 安全；不替代通用需求开发流程。
- 用户要求直接实现 AI 功能时，由 `requirement-development` 主导交付，本 skill 只补充 AI 专项约束。
- 涉及 OpenAI、Claude、Gemini、Embedding、语音、翻译、向量库或 SDK 版本时，由 `third-party-integration` 主导官方契约核验。
- 涉及服务端 API、鉴权、流式响应、幂等、任务、监控或上线策略时，结合 `backend-production-engineering`。
- 涉及模型调用日志、Token 成本、RAG 命中、工具调用链路、评测标识、Prompt/响应隐私或告警时，结合 `observability-logging-engineering`。
- 涉及向量表、检索索引、数据回填或知识库迁移时，结合 `database-performance-migration`。
- AI 行为异常、模型输出错误或 RAG 命中异常按 `fix-bugs` 确认根因，再回到本 skill 调整 AI 链路。

## 适用场景

- 新增或改造 AI Chat、智能问答、摘要、翻译、语音识别、文本生成、分类、抽取、审核、推荐等能力。
- 构建 RAG：文档切分、Embedding、向量检索、重排序、引用来源、答案生成和命中评估。
- 构建 Agent：任务拆解、工具选择、工具参数、执行状态、循环控制、失败恢复和安全边界。
- 设计 Prompt：系统提示词、Few-shot、结构化输出、JSON Schema、语言风格、拒答策略和版本管理。
- 接入或切换模型供应商：OpenAI、Anthropic、Google、Azure、火山、阿里云、向量库、语音或翻译服务。
- 优化 AI 能力：降低延迟、控制成本、提升准确率、减少幻觉、提高稳定性和可观测性。

## 工作流程

### 1. 明确目标和成功标准

先回答：

- 这个 AI 能力要帮助用户完成什么任务。
- 输入、输出、调用方和用户可见结果是什么。
- 什么结果算成功，什么结果必须拒绝或降级。
- 是否需要引用来源、结构化字段、可解释原因或人工审核。
- 是否有成本、延迟、语言、隐私、合规、并发或上线约束。

如果需求只是“加个 AI 能力”，先收敛为可验证的最小闭环，不要直接堆复杂 Agent。

### 2. 阅读现有实现和版本

修改前检查：

- 当前项目语言、框架、运行时、依赖版本和锁文件。
- 已有 AI client、HTTP client、配置读取、日志、异常、重试、缓存和测试模式。
- Prompt、模型名、API key、endpoint、超时、重试、流式响应和环境变量的既有约定。
- 是否已有 RAG、向量库、文档处理、消息历史、会话、任务或工具调用实现。

涉及第三方 API、SDK、模型或云服务时，同时使用 `third-party-integration`，按官方文档确认版本、认证、参数、错误码、限流和弃用状态。

### 3. 选择架构和边界

按任务复杂度选择最小可行方案：

- 固定规则能解决时，不要引入模型。
- 单轮生成能解决时，不要上 Agent。
- 上下文数据很少时，不要先做完整 RAG。
- 需要可追溯事实时，优先 RAG、引用来源和置信边界。
- 需要执行外部动作时，工具调用必须有参数校验、权限检查、幂等和审计。
- 需要多步骤任务时，Agent 要限制最大轮次、工具范围、超时和失败出口。

明确分层：

- 输入清洗和权限检查。
- 上下文组装、检索或工具定义。
- Prompt 或消息模板。
- 模型调用和流式处理。
- 结构化解析、校验、后处理。
- 业务落库、事件、日志和返回。

### 4. 实现 AI 链路

实现时遵守：

- 不硬编码密钥、生产 endpoint、模型名或用户隐私数据。
- 配置默认值要安全，超时、重试、最大 Token 和并发限制要可控。
- 结构化输出优先用 schema、类型或解析器校验，不依赖脆弱字符串截取。
- 流式响应要处理取消、断连、部分输出、异常收尾和资源释放。
- RAG 要保存来源信息，避免把检索不到的内容编造成事实。
- 工具调用要校验工具名、参数、权限、幂等、失败重试和副作用。
- 日志记录请求链路、模型、耗时、Token、错误类型和评测标识，但必须脱敏。

不要把 Prompt 当作唯一安全边界；涉及权限、金额、状态变更、数据删除或外部调用时，必须在代码层校验。

### 5. 评测和验证

按风险选择验证：

- 单元测试覆盖 Prompt 组装、参数校验、结构化解析、失败处理和降级逻辑。
- 使用 mock 或 fixture 覆盖模型超时、限流、空响应、非法 JSON、工具失败和检索无结果。
- 为核心能力准备最小评测集，覆盖典型问题、边界问题、不可回答问题和高风险问题。
- RAG 验证检索命中、引用来源、无命中拒答、重复文档和长文档切分。
- Agent 验证最大轮次、工具选择错误、工具异常、重复执行和权限不足。
- 生产风险较高时，提供灰度、回滚、监控指标和人工复核建议。

如果无法真实调用模型或第三方服务，说明限制，并提供可执行的本地 mock、sandbox 或 CI 验证步骤。

## 交付说明

最终说明必须覆盖：

- AI 能力结论：实现或设计了什么能力。
- 模型和第三方依据：使用的供应商、模型、SDK/API 版本和官方依据。
- 核心链路：输入、上下文、模型调用、工具/RAG、输出和降级。
- 验证结果：测试、评测、mock、手工样例或无法验证原因。
- 风险与后续：成本、延迟、幻觉、隐私、限流、监控、评测集和上线注意事项。

## 行为约束

- 不要凭记忆实现第三方 AI API 或 SDK。
- 不要把密钥、Token、用户隐私、原始音频或敏感业务数据写入代码或日志。
- 不要用 Prompt 替代权限、安全、幂等或业务规则校验。
- 不要只验证 happy path；必须覆盖空结果、模型失败、解析失败和降级。
- 不要声称 AI 效果已验证，除非有测试、评测集、样例矩阵或明确的人工验证记录。
- 不要在未确认影响范围时更换模型供应商、向量库、Embedding 模型或关键 Prompt 契约。
