# 数据库性能与迁移 Skill

## 目标

帮助 AI 编程助手安全处理数据库相关需求：模型设计正确、查询性能可控、迁移可回滚、历史数据兼容、数据修复可审计，并且不会为了短期功能埋下生产数据风险。

## 核心原则

1. 先理解业务实体、数据生命周期和读写路径，再设计字段、索引或迁移。
2. 先读取现有表结构、Mapper/ORM、查询模式、迁移工具和数据库约定，再写 SQL。
3. 涉及生产数据的改动必须考虑备份、兼容、回滚、灰度和验证。
4. 字段新增要明确默认值、空值、历史数据、索引、唯一约束和代码兼容。
5. SQL 优化要基于真实过滤条件、排序、分页、数据量和执行计划，不凭感觉加索引。
6. 事务、锁、批量更新和数据修复要评估并发、死锁、长事务和部分失败。
7. 数据库变更要与应用发布顺序匹配，避免新旧代码同时运行时不兼容。
8. 不直接对生产数据做不可逆操作；危险操作必须有演练和回滚说明。

## 与其他 Skill 的关系

- 本 skill 负责数据库结构、SQL、索引、迁移、数据修复和数据安全边界；不替代服务端业务流程设计。
- 功能开发中的数据库改动由 `requirement-development` 主导交付，本 skill 只补齐数据兼容、性能、迁移和回滚约束。
- 数据问题修复由 `fix-bugs` 主导根因和复现，本 skill 只负责安全的数据修复、回滚和验证方案。
- 应用层事务、幂等、缓存、消息和接口兼容由 `backend-production-engineering` 负责，本 skill 只处理数据库侧事实和风险。
- 向量数据库、搜索索引或第三方数据库服务涉及官方 API/SDK 时，结合 `third-party-integration`；知识库/RAG 数据链路再结合 `ai-application-development`。

## 适用场景

- 新增或修改表、字段、枚举、状态、索引、唯一约束、外键或分区。
- 优化慢 SQL、分页查询、统计查询、批量查询、N+1 查询和锁等待。
- 编写数据库迁移脚本、数据订正脚本、历史数据补齐脚本和回滚脚本。
- 处理事务隔离、并发更新、重复插入、乐观锁、悲观锁和唯一冲突。
- 调整 Redis、Elasticsearch、向量数据库、缓存表、搜索索引或统计表的数据结构。
- 排查数据不一致、脏数据、重复数据、丢数据、迁移失败和线上性能问题。

## 工作流程

### 1. 明确数据目标和影响范围

先确认：

- 涉及哪些实体、表、字段、索引、缓存或搜索索引。
- 数据如何产生、读取、更新、删除、归档和展示。
- 当前数据量、增长速度、核心查询条件和排序方式是什么。
- 是否存在历史数据、旧客户端、旧代码、报表、导出、定时任务或第三方依赖。
- 是否需要在线迁移、灰度发布、双写、回填、重建索引或停机窗口。

不清楚数据量或查询路径时，先从代码和已有 SQL 推断，并把关键不确定项列为风险。

### 2. 阅读现有数据库实现

修改前检查：

- 建表脚本、迁移目录、实体类、Mapper/Repository、XML SQL、查询构造器和测试。
- 现有命名、字段类型、时间字段、逻辑删除、租户/用户隔离、状态枚举和审计字段。
- 已有索引、唯一约束、联合查询、排序字段、分页方式和慢查询相关代码。
- 迁移工具：Flyway、Liquibase、Maven/Gradle 脚本、手工 SQL 或项目约定。
- 是否有读写分离、分库分表、缓存、搜索索引、异步同步或数据修复流程。

不要只改实体类或只改 SQL；检查数据库结构和应用模型是否一致。

### 3. 设计表结构、索引和迁移

设计时明确：

- 字段类型、长度、精度、字符集、是否允许空、默认值和注释。
- 索引目的：过滤、排序、唯一性、关联查询或覆盖查询。
- 联合索引顺序要匹配高频查询的等值条件、范围条件和排序。
- 唯一约束要考虑历史重复数据、软删除、租户隔离和并发插入。
- 大表字段新增、索引创建和数据回填要考虑锁表、批次、限速和失败续跑。
- 应用兼容顺序：先加兼容字段，再发布兼容代码，再回填，再收紧约束或删除旧字段。

不要为了一个低频查询盲目增加索引；索引会影响写入、存储和维护成本。

### 4. 实现数据库改动

实现时遵守：

- 迁移脚本命名、位置和格式遵循项目约定。
- 新字段在代码层处理空值和历史数据，不假设数据库立刻回填完成。
- 批量数据修复要分页、限速、可重入、可记录进度，避免长事务和一次性全表更新。
- 删除字段、删除索引、收紧非空或唯一约束前，先确认所有旧代码和历史数据已兼容。
- SQL 查询要限制返回范围，避免无条件全表扫描、深分页和无界导出。
- Redis、搜索索引和向量库改动要明确同步策略、重建方式、失败重试和一致性窗口。
- 数据修复脚本要输出影响行数、异常记录和可复核结果，避免静默修改。

涉及不可逆或高风险操作时，先提供方案和风险说明，不直接执行。

### 5. 性能分析和验证

按风险选择验证：

- 编译或类型检查，确认实体、Mapper、SQL 和迁移脚本语法可用。
- 单元或集成测试覆盖新增字段、查询、分页、排序、唯一冲突和历史空值。
- 对关键 SQL 使用 `EXPLAIN` 或项目可用的执行计划工具验证索引命中和扫描范围。
- 数据迁移验证影响行数、抽样结果、幂等重跑、失败恢复和回滚脚本。
- 大表或线上数据变更要提供预估耗时、锁影响、批次大小、暂停/恢复策略和监控指标。
- 涉及缓存或搜索索引时验证重建、失效、回填和最终一致性。

无法连接数据库或无法运行迁移时，必须说明限制，并给出可执行 SQL、测试命令或上线检查清单。

## 交付说明

最终说明必须覆盖：

- 数据库结论：改了哪些表、字段、索引、查询、脚本或数据。
- 兼容策略：历史数据、旧代码、旧客户端、默认值、空值和发布顺序。
- 性能依据：关键查询、索引设计理由、执行计划或无法验证原因。
- 迁移与回滚：迁移步骤、回填方式、回滚方案、影响行数和风险。
- 验证结果：测试、SQL 检查、脚本演练或无法验证原因。

## 行为约束

- 不要凭感觉加索引或改字段类型；必须结合查询路径和数据语义。
- 不要在未确认历史数据前收紧非空、唯一约束或删除字段。
- 不要写不可重入、无日志、无批次、无回滚的数据修复脚本。
- 不要忽略应用发布顺序和新旧代码兼容。
- 不要把生产数据、账号、手机号、邮箱、Token 等敏感值写入日志或示例。
- 不要声称迁移安全，除非说明了验证、备份、回滚、监控和残余风险。
