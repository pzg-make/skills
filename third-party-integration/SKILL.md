# Third Party Integration Skill

## 目标

在接入第三方 API、SDK、云服务或开源库时，避免凭记忆、过期示例或不匹配版本实现。必须基于官方文档和项目实际依赖版本确认对接方式，再进行设计、编码、测试和交付说明。

## 核心原则

1. 优先查官方文档、官方 API reference、官方 SDK 文档、官方 changelog 或 release notes。
2. 先确认项目当前使用的库版本、运行时版本和锁文件，再选择语法和 API。
3. 不要默认使用最新版文档；除非用户要求升级，否则以项目已选版本或目标版本为准。
4. 优先沿用当前项目已有实现、封装、供应商、SDK、库和版本；不要主动替换，除非当前实现确认不适配。
5. 如果当前实现与目标能力、官方契约或所选版本不适配，必须说明不适配原因、可选方案和影响，并让用户确认后再更换供应商、库、SDK、API version 或大版本。
6. 新功能或新模块需要新增第三方库时，先评估与当前语言、框架、运行时、构建工具和已有依赖的兼容性，再评估维护活跃度、稳定性、安全性和生产可用性；在满足兼容性和稳定性的前提下，尽量选择稳定的新版本。
7. 不要把博客、问答、社区示例当作最终依据；它们只能作为线索，最终必须回到官方来源确认。
8. 任何认证、权限、限流、计费、回调、安全、数据格式和错误处理都必须按官方契约实现。
9. 版本存在差异时，必须说明所选版本、差异点和兼容风险。
10. 如果无法联网或无法访问官方文档，必须明确说明无法完成官方核验，并基于本地依赖和已有代码谨慎推断。
11. 最终交付必须说明参考了哪些官方来源、适配了哪个版本、如何验证。

## 与其他 Skill 的关系

- 本 skill 主导第三方契约、版本、认证、错误码、限流、弃用和 SDK/API 语法确认；业务落地仍由对应开发或修复流程负责。
- 新功能开发由 `requirement-development` 主导，本 skill 只提供第三方接入约束。
- Bug 或版本不兼容由 `fix-bugs` 主导根因，本 skill 只提供官方依据和版本差异判断。
- AI 供应商、Embedding、语音、翻译、向量库或模型 SDK 同时使用 `ai-application-development`，由 AI skill 决定产品链路和评测边界，本 skill 决定官方接入契约。
- 服务端接口、回调、Webhook、幂等和上线风险同时结合 `backend-production-engineering`。
- 第三方调用日志、错误分类、耗时、限流、告警和敏感响应脱敏同时结合 `observability-logging-engineering`。

## 适用场景

- 调用第三方 REST、GraphQL、gRPC、WebSocket、Webhook 或文件上传 API。
- 接入 OAuth、SSO、支付、短信、邮件、推送、地图、云存储、AI、语音、翻译、搜索、监控等外部平台。
- 使用或升级第三方库、SDK、CLI、插件、框架扩展。
- 处理依赖版本升级、API 弃用、方法签名变化、配置项变化、包名迁移。
- 解决“代码示例和当前版本不一致”“编译找不到方法”“运行时报协议错误”等版本适配问题。

## 工作流程

### 1. 识别第三方能力和集成边界

先明确：

- 第三方服务、SDK 或库的名称。
- 要使用的具体能力：接口、方法、事件、回调、模型、插件或配置项。
- 当前项目语言、框架、运行时、构建工具和部署环境。
- 是新增接入、修复已有接入、升级版本，还是替换供应商。
- 是否涉及生产账号、密钥、计费、隐私、合规或安全风险。

如果需求里没有明确第三方名称或目标能力，先从代码、依赖文件、报错和上下文推断；关键不确定时再询问用户。

### 2. 确认当前版本和目标版本

实现前必须检查版本来源：

- Java：`pom.xml`、Gradle 文件、dependency management、lockfile、实际 dependency tree。
- Node.js：`package.json`、lockfile、workspace 配置、运行时版本。
- Python：`requirements.txt`、`pyproject.toml`、lockfile、虚拟环境已安装版本。
- Go/Rust/.NET/PHP 等：对应模块文件和 lockfile。
- Docker、CLI、云服务：镜像标签、CLI 版本、API version、区域和环境配置。

判断版本策略：

- 如果项目已有版本，默认适配已有版本。
- 如果新功能或新模块需要新增依赖，先检查项目语言/框架/运行时/构建工具/已有依赖兼容性，再检查官方推荐版本、长期支持状态、维护活跃度、已知漏洞、breaking changes 和生产稳定性；在这些条件满足时，尽量选择稳定的新版本，并说明选择理由。
- 如果用户明确指定新增依赖版本，仍需检查该版本的兼容性和稳定性；发现风险时先说明，不要直接使用。
- 如果用户要求升级，先查 breaking changes、迁移指南和兼容影响。
- 如果官方文档只有最新版，必须寻找目标版本文档、历史版本文档、release notes 或源码标签。
- 如果当前版本或当前封装不支持目标能力，先评估能否在现有实现上兼容扩展；只有确认无法适配时，才提出更换方案，并等待用户确认。

### 3. 查询官方资料

需要联网时，优先查询：

- 官方文档、API reference、SDK reference、guide、quickstart。
- 官方 GitHub/GitLab 仓库、README、examples、release notes、changelog、migration guide。
- 官方包管理页面和版本元数据。
- 官方 status、limits、pricing、security、auth、webhook、pagination、rate limit、error code 文档。

查询后记录：

- 官方链接或文档位置。
- 文档对应版本或发布日期。
- 与当前项目版本是否一致。
- 本次会用到的接口、方法、参数、响应、错误和限制。

如果是 OpenAI 相关能力，应同时遵守 OpenAI 官方文档查询流程和对应 OpenAI skill。

### 4. 设计对接方案

方案必须覆盖：

- 认证方式：API key、OAuth、JWT、签名、临时凭证、scope、token 刷新。
- 请求/响应契约：路径、方法、header、body、content type、分页、排序、过滤、文件上传。
- 错误处理：错误码、重试、超时、幂等、限流、降级、告警。
- 回调/Webhook：签名校验、幂等、重放防护、事件版本、失败重试。
- 配置和密钥：环境变量、配置中心、secret 管理、日志脱敏。
- 兼容性：API 版本、SDK 版本、运行时版本、依赖冲突、弃用接口。
- 测试方式：mock、sandbox、官方测试环境、契约测试、最小真实调用。

不要只实现 happy path。

### 5. 编码实现

编码时遵守：

- 使用所选版本支持的语法、类名、方法签名、配置项和错误类型。
- 优先复用项目已有 HTTP client、SDK wrapper、配置读取、日志、异常、重试和监控模式。
- 优先在当前供应商、当前 SDK/库和当前封装上扩展；需要替换时先获得用户确认。
- 不要把密钥、token、账号、区域、endpoint、模型名等敏感或环境相关值硬编码。
- 对外部响应做结构化解析和边界校验，不要依赖脆弱字符串拼接。
- 将第三方调用封装在清晰边界内，避免把 SDK 类型扩散到业务核心层。
- 记录必要日志，但避免输出密钥、个人数据、原始音频/文件内容或敏感响应。

如果官方示例与项目版本或框架不匹配，必须按项目版本调整，不能直接照抄。

### 6. 验证和回归

按风险选择验证方式：

- 编译或类型检查，确认所选版本的 API 可用。
- 单元测试覆盖成功、失败、超时、限流、空响应、字段缺失、异常格式。
- 对 HTTP/API 调用使用 mock server、fixture 或 sandbox 环境。
- 对 Webhook 验证签名、幂等、重复投递、乱序和失败重试。
- 对 SDK 升级运行相关模块测试，并检查 breaking changes 是否覆盖。
- 对生产风险较高的对接，提供灰度、回滚、监控和告警建议。

无法真实调用第三方服务时，必须说明限制，并给出用户或 CI 可执行的验证步骤。

### 7. 交付说明

最终输出按复杂度调整，但必须覆盖：

- 对接结论：接入了哪个第三方能力。
- 官方依据：查询了哪些官方文档或版本资料。
- 版本适配：当前项目版本、目标版本、语法或接口差异。
- 新增依赖理由：如果引入新库，说明兼容性、稳定性和版本选择依据。
- 修改内容：关键文件、配置、接口、封装和错误处理。
- 验证结果：运行了哪些测试或检查。
- 风险与后续：限流、计费、权限、密钥、兼容性、回调、上线或监控风险。

推荐格式：

```text
## 对接结论

说明本次接入或适配了什么第三方能力。

## 官方依据

- 官方文档/版本资料：链接或来源说明
- 适配版本：版本号或 API version

## 修改内容

- 文件 A：核心改动
- 文件 B：核心改动

## 验证结果

- 命令或方式：通过/失败

## 风险与注意事项

说明密钥、限流、计费、兼容性、上线、回调或监控风险。
```

## 行为约束

- 不要凭记忆实现第三方 API 或 SDK。
- 不要使用与项目依赖版本不匹配的官方示例。
- 不要在未获用户确认时替换当前供应商、SDK、库、API version 或大版本。
- 不要把社区答案当作官方契约。
- 不要跳过认证、限流、错误码、超时、重试和幂等处理。
- 不要硬编码密钥、token、endpoint 或生产账号。
- 不要忽略弃用、breaking changes、区域差异和 API version。
- 不要声称已按官方文档实现却没有说明官方依据。
