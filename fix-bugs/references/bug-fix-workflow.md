# Bug Fix Workflow Reference

Read this file for complex bugs, uncertain root causes, production incidents, flaky failures, broad impact changes, or when the final response needs a structured template.

## Short Loop

```text
现象 -> 证据 -> 假设 -> 最小复现 -> 根因 -> 修复层级 -> 验证矩阵 -> 交付结论
```

## 修复前闸门

除非是非常明确的小改动，修改前先确认：

- 已区分用户可见现象、日志报错、直接触发点和真正根因。
- 至少有一个可执行复现、失败测试、请求样例、日志证据或代码级推导。
- 已列出 1-3 个候选根因，并用代码、测试、日志或运行结果排除明显错误假设。
- 已确认修复层级：输入校验、状态流、业务规则、接口契约、公共工具、数据模型、并发时序或错误处理。
- 已知道修复后要跑哪些验证，验证缺失时要补什么测试或说明什么限制。

如果条件缺失，先补证据、缩小范围或询问真正阻塞的信息。

## 常见返工信号

遇到这些信号时，重新确认根因：

- 只修空指针、异常捕获或默认值，却没有解释为什么值为空。
- 只让当前测试通过，但同字段、同状态或同接口还有其他入口。
- 修改公共方法、返回结构、状态顺序或缓存行为，却未检查调用方。
- 复现依赖特定环境、账号、时间、并发或历史数据，本地验证没有覆盖。
- 偶现、时序、数据污染或缓存不一致问题被按普通同步逻辑处理。
- 用户描述业务不符合预期，但修复只围绕技术报错。

## 修复层级决策

- 单个判断、映射、参数传递、边界条件或异常处理错误，优先局部修复。
- 局部补丁只能隐藏现象，或同类输入仍会失败时，上移到抽象、契约、数据模型、状态流或公共错误处理层。
- 调整公共方法、接口契约或数据结构时，检查调用方、兼容性、测试覆盖和迁移风险。
- 根因修复会演变成大规模重构时，优先做能消除当前根因的兼容修复，并把结构性问题作为风险或后续任务说明。
- 多个方案都可行时，选择长期行为最正确、验证成本可承受、回滚路径清楚的方案。

方案自检：

- 同样输入从另一个入口进来，这个修复还成立吗？
- 字段为空、重复、乱序、超时或部分失败时，这个修复还成立吗？
- 调用方依赖旧行为时，是否需要兼容处理或迁移说明？
- 测试只覆盖当前 case 时，是否还需要补更一般的边界测试？
- 修复失败时，是否有清晰回滚方式或风险说明？

## 验证矩阵

按风险选择：

- 原始复现：同样输入、步骤或失败测试必须不再触发问题。
- 相邻边界：空值、缺失字段、重复请求、非法状态、权限不足、超时、重试、并发或历史数据。
- 相似入口：同一函数的其他调用方、同一字段的其他使用位置、同一状态流的其他分支。
- 回归范围：相关模块测试、公共逻辑测试、接口契约测试或构建检查。
- 副作用检查：日志、缓存、事务、幂等、性能、兼容性、错误返回和用户可见文案。

记录验证了什么，也记录没有验证什么。

## 测试补充

新增测试优先覆盖：

- 正常场景。
- Bug 复现场景。
- 边界场景。
- 空值或缺失字段。
- 异常输入。
- 多语言、多类型、多状态场景。
- 相似逻辑场景。

测试命名示例：

```text
should_return_original_text_when_source_and_target_language_are_same
should_not_crash_when_translation_result_is_empty
should_preserve_existing_state_when_retry_fails
```

## 类似问题排查

检查范围：

- 相同函数的其他分支。
- 相似工具类、接口、数据结构。
- 相同字段的其他使用位置。
- 相同异常处理模式。
- 相同并发或异步流程。
- 相同配置读取逻辑。

不要为了“顺手”扩大修改范围。只有确认同类问题真实存在，才额外修复；影响范围大于本次任务时，记录为后续风险。

## 输出模板

复杂 Bug 推荐：

```text
## 问题结论

简要说明这个 Bug 是什么，是否已修复。

## 复现或证据

说明复现步骤；如果无法复现，说明确认问题的证据链。

## 根因分析

说明真正原因，不只描述报错。

## 修复方案

说明修改了哪些逻辑，以及为什么选择这个修复层级。

## 修改内容

- 文件 A：修改点
- 文件 B：修改点

## 测试验证

- 测试命令 1：通过/失败
- 测试命令 2：通过/失败

## 类似问题排查

说明检查了哪些相似逻辑，是否发现同类问题。

## 风险与注意事项

说明兼容性、性能、数据、并发或历史行为风险。

## 最终结论

确认 Bug 是否已修复。
```

简单 Bug 可压缩为：

```text
已修复：一句话说明问题和根因。

改动：列出关键文件和逻辑。

验证：列出测试或无法测试的原因。

风险：说明是否有相似问题或残余风险。
```
