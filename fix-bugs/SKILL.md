# Bug Fix Skill

## 目标

根据用户输入的问题描述、报错信息、截图、日志、代码片段或项目上下文，系统性完成 Bug 定位、原因确认、以根因为边界的最优修复、回归验证，并输出清晰的修复方案与复现步骤。

## 核心原则

1. 优先复现，再修复；无法完整复现时，必须说明证据链和不确定点。
2. 先理解业务预期，再判断代码对错。
3. 优先采用“最优解中的最小必要改动”：先找能真正消除根因的正确方案，再把改动控制在该方案所需的最小范围内。
4. 必须确认根因，而不是只消除表面报错。
5. 修改后必须跑原有测试，并补充必要的新测试。
6. 必须评估相似问题，避免只针对当前输入过度拟合。
7. 按风险和复杂度调整流程粒度：小问题保持轻量，复杂问题明确记录取舍、影响面和验证边界。
8. 输出内容必须覆盖：复现或证据、根因、修改点、验证方式、风险点；不强制套固定大模板。

## 输入信息

用户可能提供以下一种或多种信息：

- Bug 描述、预期行为、实际行为、操作步骤。
- 报错日志、截图、相关代码、项目目录。
- 测试用例、接口请求与响应。
- 线上环境信息、版本号、分支名、提交记录。

如果信息不足，不要立即假设结论。应优先基于已有项目代码和日志进行推断；只有关键阻塞信息缺失时，才询问用户。

## 工作流程

### 1. 理解问题与业务预期

先明确：

- 用户遇到的具体问题是什么。
- 正确行为应该是什么。
- 当前错误行为是什么。
- 问题出现在哪个模块、页面、接口或流程。
- 是必现、偶现，还是特定条件下出现。
- 是否与最近改动、配置、环境、数据有关。

输出初步判断：问题类型、可能影响范围、初步排查方向。

### 2. 结合项目定位问题

基于项目代码进行定位，不要只看用户贴出的局部代码。优先检查：

- 入口函数、调用链路、参数传递、状态变更。
- 配置读取、异常处理、数据库读写、缓存逻辑。
- 并发/异步逻辑、权限判断、边界条件。
- 测试失败位置、最近相关提交。

需要回答：

- Bug 出现在哪个文件。
- 关键函数/类是什么。
- 错误逻辑在哪里。
- 上游输入是什么。
- 下游影响是什么。

### 3. 复现 Bug

在修改代码前，尽量先复现问题。复现方式包括：

- 运行已有测试用例。
- 手动构造输入。
- 根据日志还原请求。
- 使用最小复现 case。
- 添加临时断言或日志。
- 本地启动相关服务验证。

复现时记录：复现步骤、输入条件、实际结果、报错信息、触发原因。

如果无法完整复现，也要说明当前能够确认的证据和无法复现的原因。

### 4. 确认根因

不要把报错信息直接当作根因。需要区分：

- 表面现象。
- 直接原因。
- 根本原因。

根因分析应说明：

- 哪个条件触发了问题。
- 哪段代码处理不正确。
- 为什么之前没有暴露。
- 为什么当前输入会失败。
- 是否存在历史设计缺陷。
- 是否存在数据、配置、并发、边界值问题。

示例：

```text
根因：
在 xxx 场景下，yyy 字段可能为空，但 zzz 函数默认认为它一定存在，
导致空指针/错误分支/数据覆盖。该问题在普通场景下不会出现，
因为 aaa 数据通常会被提前初始化。
```

### 5. 制定修复方案

修复方案必须遵循：

- 最优解中的最小必要改动。
- 不破坏现有行为。
- 不引入无关重构。
- 不硬编码当前 case。
- 不只针对单个输入打补丁。
- 保持代码风格一致。
- 保持已有接口兼容。
- 保持错误处理清晰。

如果 Bug 来自原架构、模块边界或历史设计的局限，不要为了追求表面上的“最小改动”而堆叠局部补丁。应比较可行方案，选择能真正解决根因、长期行为正确、风险可控的方案；然后只实施该方案必须涉及的最小文件、接口和测试改动。

#### 修复层级决策

当存在多个修复层级时，按以下边界选择：

- 如果错误仅来自单个判断、映射、参数传递、边界条件或异常处理，优先局部修复。
- 如果局部补丁只能隐藏现象，或同类输入仍会失败，必须上移到抽象、契约、数据模型、状态流或公共错误处理层修复。
- 如果公共方法、接口契约或数据结构需要调整，必须检查调用方、兼容性、测试覆盖和迁移风险。
- 如果根因修复会演变成大规模重构，优先做能消除当前根因的兼容修复，并把剩余结构性问题作为风险或后续任务说明。
- 如果存在多个可行方案，选择长期行为最正确、验证成本可承受、回滚路径清楚的方案，而不是代码行数最少的方案。

制定方案时应显式权衡：

- 局部补丁是否只隐藏现象。
- 是否需要调整抽象、调用边界、数据模型、状态流或错误处理策略。
- 较深层修复是否会触碰公共契约、迁移数据或影响调用方。
- 是否可以分阶段处理：本次修复根因，后续重构另行记录。

修复前应说明：准备修改哪些文件、修改原因、是否需要新增测试、是否存在兼容性风险。

### 6. 修改 Bug

修改代码时优先选择：

- 修正错误判断、边界处理、参数传递。
- 修正状态更新顺序、异常处理、异步/并发时序。
- 修正数据映射、类型转换、配置读取、测试断言。

避免：

- 脱离根因的大范围重构。
- 删除测试绕过问题。
- 简单 try-catch 吞掉异常。
- 为了通过当前测试写死逻辑。
- 修改与 Bug 无关的功能。
- 改动公共方法但不评估调用方。

### 7. 跑原有测试

修改后必须优先运行与 Bug 相关的已有测试。如果项目支持，应按顺序执行：

```text
1. 运行失败用例或相关模块测试。
2. 运行当前模块测试。
3. 在条件允许时运行全量测试。
```

需要记录：

- 执行了哪些测试。
- 测试是否通过。
- 如果失败，失败原因是什么。
- 是否与本次修复有关。

### 8. 补充或更新测试用例

如果原有测试没有覆盖该 Bug，应新增测试。测试应覆盖：

- 正常场景。
- Bug 复现场景。
- 边界场景。
- 空值/缺失字段。
- 异常输入。
- 多语言/多类型/多状态场景。
- 相似逻辑场景。

测试命名应清楚表达场景，例如：

```text
should_return_original_text_when_source_and_target_language_are_same
should_not_crash_when_translation_result_is_empty
should_preserve_existing_state_when_retry_fails
```

### 9. 检查类似问题，避免过度拟合

修复当前 Bug 后，需要主动搜索类似模式。检查范围包括：

- 相同函数的其他分支。
- 相似工具类、接口、数据结构。
- 相同字段的其他使用位置。
- 相同异常处理模式。
- 相同并发/异步流程。
- 相同配置读取逻辑。

需要判断：

- 是否只有当前位置有问题。
- 是否存在同类隐藏 Bug。
- 是否需要统一修复。
- 是否应该抽出公共校验逻辑。
- 是否只需要记录风险，不做额外修改。

不要为了“顺手”扩大修改范围。只有确认同类问题真实存在，才进行额外修复。

### 10. 确认修复

最终确认标准：

- Bug 可以稳定复现，或有充分证据说明原始问题的触发条件。
- 修复后复现步骤不再触发问题。
- 原有测试通过。
- 新增测试通过。
- 相似问题已排查。
- 没有引入明显副作用。
- 代码改动范围合理。
- 输出结论清晰。

## 项目验证补充

当另一个项目专用 skill 可用时，同时遵守该项目的构建、测试、代码风格和验证命令要求。不要在本通用 skill 中假设特定仓库、模块名、构建工具路径或部署环境。

## 输出格式

每次修复完成后，按问题复杂度输出。小问题可以用短结论；复杂问题使用结构化说明。无论格式长短，都必须覆盖以下信息：

- 问题结论：这个 Bug 是什么，是否已修复。
- 复现或证据：如何复现；如果无法复现，说明确认问题的证据链。
- 根因分析：真正原因是什么，不只描述报错。
- 修复方案：改了哪些逻辑，为什么选择这个层级的修复。
- 修改内容：列出关键文件和核心改动。
- 测试验证：执行了哪些测试，结果如何；如果无法执行，说明原因和建议命令。
- 类似问题排查：检查了哪些相似逻辑，是否发现同类问题。
- 风险与注意事项：兼容性、性能、数据、并发、历史行为或回滚风险。

复杂 Bug 推荐使用：

```text
## 问题结论

简要说明这个 Bug 是什么。

## 复现步骤

1. ...
2. ...
3. ...

## 根因分析

说明真正原因，不只描述报错。

## 修复方案

说明修改了哪些文件、哪些逻辑，以及为什么这样改。

## 修改内容

- 文件 A：修改点
- 文件 B：修改点

## 测试验证

已执行：

- 测试命令 1：通过/失败
- 测试命令 2：通过/失败

新增测试：

- 测试用例名称：覆盖场景

## 类似问题排查

说明检查了哪些相似逻辑，是否发现同类问题。

## 风险与注意事项

说明是否存在兼容性、性能、数据、并发或历史行为风险。

## 最终结论

确认 Bug 是否已修复。
```

简单 Bug 可以压缩为：

```text
已修复：一句话说明问题和根因。

改动：列出关键文件和逻辑。

验证：列出测试或无法测试的原因。

风险：说明是否有相似问题或残余风险。
```

## 行为约束

- 不要在没有证据的情况下直接修改代码。
- 不要只根据报错信息下结论。
- 不要跳过复现步骤。
- 不要为了通过测试删除断言。
- 不要扩大无关修改。
- 不要把临时日志、调试代码留在最终提交中。
- 不要只修当前输入，必须检查相似问题。
- 如果无法运行测试，必须明确说明原因，并给出建议的验证命令。
- 如果无法确认根因，必须说明当前证据和不确定点。

## 推荐执行顺序

```text
理解问题
→ 定位代码
→ 复现 Bug
→ 确认根因
→ 制定最优解中的最小必要修复方案
→ 修改代码
→ 跑原有测试
→ 新增/更新测试
→ 检查类似问题
→ 确认修复
→ 输出复现步骤、根因、修复方案和验证结果
```
