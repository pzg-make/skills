# Requirement Development Skill

## 目标

根据用户输入的需求描述、业务背景、设计稿、接口说明、代码上下文、反馈记录或技术目标，完成从需求理解到代码交付的端到端开发闭环：明确范围、选择方案、实施改动、补充验证、说明风险，并输出可交付的结果。

## 核心原则

1. 先理解用户要达成的业务或技术结果，再决定实现方式。
2. 先读现有代码和项目约定，再新增抽象、接口、字段或依赖。
3. 优先做“满足当前目标的最小必要实现”，但不牺牲长期正确性、可维护性和安全性。
4. 需求不清时先基于上下文推进；只有关键决策会影响方向、数据或兼容性时才询问用户。
5. 保持改动聚焦，不夹带无关重构、格式化或元数据变更。
6. 任何用户可见行为、接口契约、数据结构或共享逻辑变更，都必须评估调用方、兼容性和测试。
7. 实现后必须验证；无法验证时说明原因、影响和建议命令。
8. 最终交付要说明做了什么、如何验证、还有什么风险。

## 与其他 Skill 的关系

- 需求仍模糊、需要拆范围或排期时，由 `requirement-breakdown` 主导，先拆清楚再开发。
- 需求本质是 Bug、回归或异常行为时，由 `fix-bugs` 主导根因修复和回归验证。
- 涉及生产级后端接口、鉴权、幂等、缓存、消息、任务、监控或上线策略时，本 skill 负责端到端落地，`backend-production-engineering` 只补充服务端生产化边界。
- 涉及表结构、索引、慢 SQL、迁移、数据修复或历史数据兼容时，结合 `database-performance-migration` 补充数据库风险。
- 涉及 RAG、Agent、Prompt、模型调用、AI 评测或 AI 安全边界时，结合 `ai-application-development` 补充 AI 能力质量和生产化约束。
- 涉及第三方 API、SDK、云服务、模型供应商或依赖版本时，由 `third-party-integration` 主导官方契约和项目版本确认。
- 涉及日志、Trace、指标、告警、审计或脱敏时，结合 `observability-logging-engineering`，只在关键步骤补充必要观测。
- 当前项目存在项目专用 skill 时，同时遵守该项目的模块边界、构建命令、测试约定、代码风格和交付规则。

## 输入信息

用户可能提供以下一种或多种信息：

- 功能需求、业务目标、技术目标、用户反馈、验收要求。
- 原型、截图、接口文档、数据结构、错误日志、测试用例。
- 项目目录、相关模块、代码片段、历史实现、设计约束。
- 多端范围、第三方服务、上线时间、兼容性或安全要求。

如果输入不足，先从代码和现有模式中补齐上下文，并把不确定项列为假设。只有以下情况必须询问用户：

- 存在多个互斥业务规则，且无法从代码或上下文判断。
- 需要破坏兼容性、迁移数据、改变公开接口或删除旧能力。
- 涉及权限、安全、计费、合规、生产数据或上线策略的关键决策。
- 继续实现会明显偏离用户目标或造成较大返工风险。

## 工作流程

### 1. 理解需求和成功标准

先明确：

- 用户真正要解决的问题是什么。
- 最终使用方、调用方或维护方是谁。
- 本次交付后，什么行为算成功。
- 是否有明确的非本期范围、兼容要求或上线约束。
- 需求属于新增、改造、集成、重构、优化、测试补充还是治理。

如果需求可以直接开发，给出简短判断后继续推进；如果需求仍过于发散，先收敛范围或列出待确认问题。

### 2. 阅读现有实现

修改前必须理解现有系统：

- 入口页面、接口、服务、任务、事件、消息或配置。
- 调用链路、状态流转、数据读写、缓存和第三方依赖。
- 现有代码风格、命名、异常处理、权限模型和测试模式。
- 类似功能的既有实现，优先复用项目已有约定。

不要只看用户贴出的局部代码，也不要为了新需求绕开已有架构。

### 3. 制定实现方案

方案必须说明：

- 准备修改哪些模块、文件或接口。
- 为什么选择这个实现层级，而不是更浅的补丁或更深的重构。
- 是否影响现有接口、数据、权限、缓存、异步任务或调用方。
- 是否需要新增配置、迁移脚本、监控日志或回滚策略。
- 是否需要新增或更新测试。

当存在多个方案时，选择长期行为正确、改动范围可控、验证路径清楚的方案。不要只按代码行数最少来判断。

### 4. 实施代码改动

实施时遵守：

- 贴合当前项目模式和模块边界。
- 对共享逻辑、公共接口、数据模型和权限行为保持谨慎。
- 新增 API 时明确请求、响应、错误、权限、幂等、分页和兼容性。
- 新增数据字段时考虑默认值、历史数据、索引、唯一约束和回滚。
- 新增异步、缓存或第三方调用时考虑超时、重试、幂等、降级和日志。
- 前端开发要覆盖加载、空态、错误态、权限态和移动/桌面适配。

避免：

- 为了赶需求硬编码当前 case。
- 复制粘贴大量相似逻辑而不复用已有能力。
- 引入未被项目使用的新框架或新依赖，除非确有必要。
- 顺手改无关代码、重排大文件或做无关格式化。
- 删除或弱化测试来让构建通过。

### 5. 测试和验证

按风险选择验证层级：

1. 优先运行与改动直接相关的测试。
2. 再运行受影响模块的测试或编译。
3. 影响公共逻辑、接口契约、数据模型或关键链路时，扩大验证范围。
4. 前端或交互改动需要实际打开页面或截图验证。
5. 数据、权限、并发、异步、缓存、第三方集成等改动需要覆盖异常路径。

新增或更新测试时，优先覆盖：

- 核心成功路径。
- 关键失败或异常路径。
- 边界值、空值、缺失字段、重复请求。
- 权限、兼容性、历史数据或多状态流转。
- 与本次需求最容易回归的逻辑。

如果无法运行测试，必须说明阻塞原因，并给出可执行的建议验证命令。

### 6. 检查影响面

交付前检查：

- 是否影响现有接口、客户端、调用方、配置、数据库或权限。
- 是否存在同类功能需要同步调整。
- 是否需要文档、接口说明、客户端接入说明或迁移说明。
- 是否存在性能、并发、安全、数据一致性或回滚风险。
- 是否留下临时日志、调试代码、测试数据或待办标记。

不要为了“顺手”扩大范围；只有确认同类改动真实必要，才同步处理。

### 7. 交付说明

最终说明按需求复杂度调整。简单需求可以简短输出；复杂需求需要结构化说明。无论格式长短，都必须覆盖：

- 需求结论：已实现什么能力或改造。
- 修改内容：关键文件、模块和核心逻辑。
- 验证结果：执行了哪些测试或检查，结果如何。
- 兼容和风险：是否影响接口、数据、权限、性能、客户端或上线。
- 后续事项：未做内容、待确认问题或建议后续优化。

复杂需求推荐使用：

```text
## 需求结论

说明本次需求是否已完成，以及交付了什么能力。

## 修改内容

- 文件或模块 A：核心改动
- 文件或模块 B：核心改动

## 验证结果

- 命令或方式 1：通过/失败
- 命令或方式 2：通过/失败

## 影响范围

说明接口、数据、权限、配置、客户端、性能或发布影响。

## 风险与后续

说明残余风险、未覆盖验证、待确认问题或建议后续任务。
```

简单需求可以压缩为：

```text
已完成：一句话说明实现结果。

验证：列出执行过的检查或无法验证原因。

注意：说明兼容性、风险或后续事项。
```

## 行为约束

- 不要跳过代码阅读直接实现。
- 不要把不明确的业务规则写死成默认行为。
- 不要擅自破坏兼容性、删除旧数据或改变公开契约。
- 不要在没有必要时引入新依赖、新架构或大范围重构。
- 不要留下临时日志、调试代码、无用注释或测试残留。
- 不要只实现成功路径，必须评估异常、边界和权限。
- 不要声称已验证但实际没有运行或确认。
