# 服务端生产化工程 Skill

## 目标

帮助 AI 编程助手把服务端需求做成生产可用的工程交付：接口契约清楚、权限和数据边界正确、异常路径完整、并发和一致性可控、可观测、可灰度、可回滚、可验证。

## 核心原则

1. 先理解业务链路、调用方和数据状态，再设计接口和服务边界。
2. 先阅读现有架构、模块边界、异常体系、权限模型和测试模式，再新增抽象。
3. 公共接口、数据模型、权限、事务、缓存和异步任务变更必须检查调用方和兼容性。
4. 生产级代码不能只实现成功路径；必须覆盖鉴权、校验、空值、重复请求、非法状态、超时和部分失败。
5. 涉及副作用的操作要评估幂等、并发、事务边界、重复投递、回滚和审计。
6. 日志、指标、告警和错误返回要服务于排障，不泄露敏感信息。
7. 不为当前 case 硬编码，不绕开项目已有封装、配置、异常和测试约定。
8. 验证范围按影响面扩大；核心链路和共享逻辑变更必须有更充分的回归检查。

## 与其他 Skill 的关系

- 本 skill 负责服务端生产化边界，不替代 `requirement-development` 的端到端需求交付，也不替代 `fix-bugs` 的根因定位。
- 新增或改造服务端功能时，由 `requirement-development` 主导，本 skill 只补充接口、权限、幂等、事务、缓存、消息、观测和上线约束。
- 服务端问题修复时，由 `fix-bugs` 主导复现和根因，本 skill 只约束修复层级、并发一致性和生产验证。
- 涉及表结构、SQL、索引、迁移或数据修复时，结合 `database-performance-migration`，不要只从 Service 层判断数据风险。
- 涉及日志、Trace、指标、告警、审计或脱敏细节时，结合 `observability-logging-engineering`；不要为了生产化而添加不必要日志。
- 涉及第三方 API、Webhook、SDK 或云服务时，结合 `third-party-integration`；涉及 AI 服务链路时，再结合 `ai-application-development`。

## 适用场景

- 新增或修改 REST、RPC、WebSocket、Webhook、内部服务接口或后台任务。
- 设计鉴权、权限、角色、数据范围、操作审计和敏感字段处理。
- 实现订单、订阅、支付、文件、通知、设备、用户、配置、统计等带副作用的服务端能力。
- 引入或调整缓存、消息队列、定时任务、分布式锁、异步处理、重试和降级。
- 排查或治理接口超时、重复执行、数据不一致、并发冲突、缓存污染、消息堆积和稳定性问题。
- 补齐生产上线需要的日志、监控、告警、灰度、回滚和运维说明。

## 工作流程

### 1. 明确接口和调用链路

先确认：

- 调用方是谁：前端、移动端、内部服务、定时任务、回调、第三方还是运维脚本。
- 入口是什么：API、消息、任务、事件、WebSocket 或命令。
- 输入、输出、状态变化、副作用和错误返回是什么。
- 是否影响旧客户端、旧接口、历史数据、权限范围或下游服务。
- 成功、失败、重试、重复请求和超时分别应如何表现。

如果接口契约不清楚，先给出最小明确契约；影响公开 API 或客户端兼容时必须等待确认。

### 2. 阅读现有服务端模式

修改前检查：

- 路由、Controller、Service、Repository/Mapper、DTO、VO、异常、响应包装和分页模型。
- 鉴权、权限注解、数据范围、操作日志、审计和敏感字段脱敏。
- 事务注解、锁、缓存 key、消息 topic、任务调度、重试和幂等实现。
- 已有同类接口或业务流程，优先复用命名、错误码、返回结构和测试风格。
- 构建、测试、启动、配置和 CI 约定。

不要只看单个文件；至少沿入口到数据读写或外部副作用走完主链路。

### 3. 设计生产级方案

方案必须明确：

- 接口契约：路径、方法、请求体、响应体、错误码、鉴权、分页、排序和兼容要求。
- 业务规则：状态流转、权限边界、输入校验、重复请求和非法状态处理。
- 一致性策略：事务边界、锁粒度、幂等 key、唯一约束、消息顺序和补偿方式。
- 性能策略：查询范围、缓存、批量处理、分页、异步化、限流和超时。
- 可观测性：关键日志、Trace/requestId、指标、告警和排障字段。
- 发布策略：配置、灰度、开关、回滚、数据迁移和客户端接入说明。

选择方案时优先保持现有架构一致；新增依赖或跨模块抽象必须有明确收益。

### 4. 实现服务端改动

实现时遵守：

- 请求参数使用明确 DTO 或实体类，避免松散 `Map` 承载业务契约。
- 对输入做结构化校验：必填、范围、枚举、权限、状态、重复和数据归属。
- 写操作要考虑幂等、事务、并发和重复提交；不能只依赖前端防抖。
- 缓存要明确 key、TTL、穿透、击穿、失效时机和脏数据风险。
- 消息和任务要处理重复投递、乱序、失败重试、死信、超时和资源释放。
- 外部调用要设置超时、错误分类、重试条件、降级和日志脱敏。
- 错误返回要稳定、可理解，不把内部异常栈或敏感信息暴露给客户端。

避免在业务服务中混入大量协议解析、第三方 SDK 细节或临时脚本逻辑；保持边界清楚。

### 5. 验证和回归

按影响面验证：

- 编译、类型检查或项目推荐的最小构建命令。
- 单元测试覆盖参数校验、权限、状态流转、事务边界、异常路径和幂等。
- 接口测试覆盖成功、失败、缺失字段、非法枚举、无权限、重复请求和历史兼容。
- 缓存、消息、任务、锁和异步逻辑要验证重复、超时、失败重试和资源清理。
- 涉及性能时检查 SQL、分页、批量大小、索引、缓存命中和慢路径。
- 涉及上线时说明灰度、开关、回滚、监控和告警验证方式。

无法运行测试时，必须说明原因，并提供用户或 CI 可执行的完整验证命令。

## 交付说明

最终说明必须覆盖：

- 服务端结论：新增或改造了什么能力。
- 接口和链路：入口、请求响应、权限、状态变化和副作用。
- 生产化处理：幂等、事务、缓存、消息、限流、降级、日志或监控。
- 验证结果：运行了哪些测试、检查或无法验证原因。
- 影响与风险：兼容性、数据、权限、性能、上线、回滚和客户端接入注意事项。

## 行为约束

- 不要跳过权限、幂等、异常路径和兼容性评估。
- 不要直接改公共接口、返回结构、状态枚举或数据模型而不检查调用方。
- 不要用捕获异常、默认值或空判断掩盖真正的业务状态问题。
- 不要在日志中输出密钥、Token、手机号、邮箱、身份证、原始请求体等敏感信息。
- 不要为了当前需求引入新框架、新中间件或大范围重构，除非收益和风险清楚。
- 不要声称可生产上线，除非已经说明验证、监控、灰度、回滚和残余风险。
