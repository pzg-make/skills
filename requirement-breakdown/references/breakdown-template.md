# Requirement Breakdown Template

Read this file when producing a formal requirement breakdown document, writing acceptance criteria, assigning priorities, or including release and rollback details.

## 优先级规则

Use P0/P1/P2/P3:

```text
P0：核心闭环必须有，没有就无法上线或无法验证
P1：重要能力，影响主要体验、稳定性或交付质量
P2：增强能力，可以延后上线
P3：优化能力，有价值但不是当前重点
```

判断标准：

- 是否影响核心交付闭环或核心技术目标。
- 是否影响关键链路、关键数据、权限、安全、稳定性或上线。
- 是否有替代方案。
- 是否依赖其他模块。
- 是否存在法律、合规、财务、性能或数据风险。

## 验收标准写法

推荐 Given/When/Then：

```text
Given 操作方具备访问权限且前置条件已满足
When 操作方提交完整且合法的输入
Then 系统应完成目标处理，返回明确结果，并进入预期状态
```

中文格式：

```text
前置条件：相关配置已启用，操作方具备必要权限。
操作步骤：操作方提交完整且合法的请求。
预期结果：系统处理成功，状态和数据变化正确，相关日志、事件或通知符合预期。
```

技术类任务验收标准应包含编译/测试通过、关键场景验证、异常场景覆盖、监控或日志可用、兼容旧行为。

## 标准输出格式

如果某项与需求无关，标记“无”或简要说明跳过原因。

```text
# 开发需求拆分文档

## 1. 需求背景与目标

说明需求来源、要解决的问题、业务目标或技术目标。

## 2. 角色与调用方

列出涉及的人、端、系统、服务及职责。

## 3. 范围边界

### 本期范围

列出本期必须实现或改造的内容。

### 非本期范围

列出明确不做的内容。

### 假设与约束

列出当前拆分基于的假设、技术约束、时间约束或兼容要求。

## 4. 核心流程

描述主流程、分支流程、异常流程或技术执行链路。

## 5. 功能/能力拆分

按模块拆分功能点或技术能力点。

## 6. 页面/交互拆分

列出涉及页面、组件、字段、操作、状态和跳转关系。

## 7. 接口/服务拆分

列出接口、服务、参数、响应、权限、异常、幂等和事务要求。

## 8. 数据/状态拆分

列出核心实体、字段、状态、索引、迁移和兼容要求。

## 9. 权限、安全与非功能需求

列出权限矩阵、安全、性能、并发、日志、监控、兼容性等要求。

## 10. 开发任务拆分

按前端、后端、数据、测试、运维、发布等拆分任务。

## 11. 优先级

标记 P0、P1、P2、P3，并说明判断依据。

## 12. 依赖关系

说明任务之间的前后依赖和外部依赖。

## 13. 测试与验收

列出需要覆盖的测试类型、关键用例、验收标准和验证方式。

## 14. 发布与回滚

说明上线步骤、灰度策略、配置项、监控项和回滚方式。

## 15. 风险点

列出可能影响交付、质量、性能、数据或上线的风险。

## 16. 待确认问题

列出需要需求方、业务方、研发、测试、运维或外部协作方确认的问题。
```

## 依赖关系示例

- 前端依赖接口，接口依赖数据库表，接口依赖权限模型。
- 业务流程依赖状态流转，统计分析依赖事件记录或日志。
- 外部服务、通知、智能处理、文件处理依赖第三方配置。
- 数据迁移依赖字段定义、脚本、备份和回滚方案。
- 发布依赖配置、环境变量、监控、灰度和运维流程。

## 风险示例

- 业务规则不明确、范围过大、优先级冲突。
- 状态流转复杂、多端数据一致性、并发重复操作。
- 权限边界不清晰、历史数据兼容、数据迁移风险。
- 外部服务不稳定、回调或异步结果异常、超时和重试风险。
- 性能瓶颈、发布回滚困难、监控不足、测试覆盖不足。
